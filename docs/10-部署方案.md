# 部署方案

---

## 一、本地开发环境

### 前置条件

```bash
# 必须
Python >= 3.11
Node.js >= 20
Docker + Docker Compose
uv（Python 包管理器）

# 安装 uv
curl -LsSf https://astral.sh/uv/install.sh | sh
```

### 1. 启动 Milvus

```bash
# 在项目根目录
docker compose up -d

# 验证启动
curl http://localhost:9091/healthz
# 返回 {"status":"ok"} 即成功

# 查看日志
docker compose logs -f standalone
```

### 2. 启动 FastAPI 后端

```bash
# 安装依赖
cd /path/to/llm-wiki
uv sync

# 复制并填写配置
cp .env.example .env
# 编辑 .env，填写 DASHSCOPE_API_KEY

# 启动
uv run uvicorn api.main:app --reload --port 8000
```

### 3. 启动 Vue 前端（开发模式）

```bash
cd web
npm install
npm run dev
# 访问 http://localhost:5173
```

### 4. 安装 Playwright 浏览器（爬虫用）

```bash
uv run playwright install chromium
```

## 四、已确认决策

> 更新于 2026-05-29

### ✅ Electron 位置
**决策：放在 `llm-wiki/collector/` 子目录（monorepo 结构）**

理由：Electron 是整个系统的一块能力，与后端 API 强耦合，统一管理更方便。

---

### ✅ 开发顺序
**决策：先建立数据模型 + 实现 Electron 数据采集，再实现后端处理流水线**

顺序：
1. 数据模型 + 存储层（基础）
2. Electron 采集应用（产生 raw 数据）
3. FastAPI 后端（处理流水线 + 检索 API）
4. Vue 前端

理由：有真实数据（raw 文档）才能调试和验证后续处理链路。

---

### ✅ Electron 目标平台
**决策：macOS + Windows 双平台**

Electron 天然跨平台，`electron-builder` 配置两个目标即可，额外工作量很小。

---

### ✅ 登录采集方案
**决策：Electron 内嵌浏览器（推荐，全程使用此方案）**

原因：
- **手动阶段**：用户在 Electron 内嵌浏览器登录一次，Cookie 自动 persist，点「保存」即采集
- **半自动阶段**：粘贴批量 URL，Electron 用已登录 Session 依次访问
- **全自动阶段**：主进程集成 Playwright，读取 `persist:wiki-collector` 的 Cookie，实现定时无人值守采集

Chrome 插件方案在全自动化上有沙盒壁垒，长期看劣势明显。
