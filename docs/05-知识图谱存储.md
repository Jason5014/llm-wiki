# 知识图谱存储设计

> Wiki 知识空间的文件结构、格式规范和管理策略

---

## 一、三层知识空间

```
wiki/
├── source/    ← 第一层：原始文章的结构化摘要（每篇文章一个页面）
├── entity/    ← 第二层：跨文章整合的实体百科（产品/公司/人物/工具）
├── concept/   ← 第三层：技术概念百科（理论/方法/模式/术语）
└── index.md   ← 全局目录 + 别名索引 + 健康统计
```

### 三层的关系

```
source 页面引用 → entity 和 concept 页面（通过 [[WikiLink]]）
entity 页面引用 → 其他 entity 和 concept 页面
concept 页面引用 → 其他 concept 和 entity 页面

index.md 汇总所有页面，提供入口导航
```

---

## 二、WikiLink 语法规范

所有页面内部链接使用 Obsidian 兼容的 `[[WikiLink]]` 格式：

| 链接类型 | 语法 | 示例 |
|---------|------|------|
| 链接到实体 | `[[entity名称]]` 或 `[[Claude]]` | `[[Claude]]`、`[[OpenAI]]` |
| 链接到概念 | `[[concept名称]]` 或 `[[agent]]` | `[[agent]]`、`[[rag]]` |
| 链接到来源 | `[[source/doc_id]]` | `[[source/15-hermes-features]]` |
| 带显示文本 | `[[目标\|显示文本]]` | `[[agent\|智能体]]` |

**命名规范**：
- entity：PascalCase 英文名，如 `Claude`、`LangChain`、`DeepSeek`
- concept：小写英文或中文，如 `agent`、`rag`、`工具调用`
- source：doc_id（由 URL slug + hash 生成），如 `15-hermes-features_a1b2c3`

## 三、双向链接机制

Wiki 系统自动维护双向引用索引：

```python
# 当 entity/Claude.md 中有 [[agent]]
# 系统自动在 concept/agent.md 的 index 中记录
# "Claude 引用了 agent"

# 实现方式：在 index.md 中维护反向引用表
# 或在内存中构建，用于图谱可视化
```

**用途**：
- 知识图谱的边数据来源
- Wiki 浏览器中"引用了此页面"功能
- 孤立页面检测（没有任何引用的页面）

---

## 四、实体/概念命名去重规则

在抽取阶段，同一实体可能出现多种写法，需要统一：

| 问题 | 示例 | 处理方式 |
|------|------|---------|
| 大小写不一致 | `claude` vs `Claude` | 统一为 PascalCase |
| 中英文混用 | `智能体` vs `Agent` vs `agent` | 在 aliases 中记录，正名用 `agent` |
| 缩写 | `LLM` vs `大语言模型` | 正名 `llm`，aliases 包含两者 |
| 版本号 | `Claude 3.5` vs `Claude` | 具体版本号归入 `Claude` 实体 |

**别名索引**（在 `index.md` 中维护）：

```markdown
| 别名 | 指向 |
|------|------|
| 智能体 | [[concept/agent]] |
| 大语言模型 | [[concept/llm]] |
| 大模型 | [[concept/llm]] |
| 扣子 | [[entity/Coze]] |
```

## 五、知识库健康检查

`index.md` 末尾维护健康统计：

```markdown
## 健康状态

- 原始文档：96 篇
- Source 页面：96 个（100%）
- Entity 页面：25 个
- Concept 页面：25 个
- 孤立页面（无任何链接）：0
- 双向链接问题：0
- 上次更新：2026-05-29
```

**健康检查项**：
1. 孤立页面：entity/concept 页面中没有被任何 source 引用的
2. 断链检查：`[[WikiLink]]` 指向不存在的页面
3. 重复实体：相似度高的实体名称（提示可能需要合并）

## 六、知识更新策略

### 新增文档时

```
新 raw 文档
    ↓
生成 source 页面（新增）
    ↓
更新 entity/concept 页面（追加 source_docs + 可能更新描述）
    ↓
更新 index.md 统计
    ↓
增量更新向量索引（只添加新内容）
```

### 删除文档时

```
删除 raw 文档
    ↓
删除对应 source 页面
    ↓
检查 entity/concept 的 source_docs 引用
    ↓
如果某 entity/concept 的 source_docs 变为空 → 考虑删除该页面
    ↓
重建 index.md
    ↓
从 Milvus 删除对应向量（按 doc_id 过滤删除）
```

## 七、与 Obsidian 的兼容性

由于使用标准的 Markdown + `[[WikiLink]]` 格式，知识库完全兼容 Obsidian：

1. 将 `wiki/` 目录作为 Obsidian Vault 打开
2. 可以使用 Obsidian 的图谱视图查看关系
3. 可以手动编辑页面，内容会被系统读取
4. 不影响系统的自动处理流程

> **注意**：手动在 Obsidian 中编辑后，需要重新触发「重建索引」操作更新向量库和 FTS5。
