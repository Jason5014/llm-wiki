# 数据采集层设计方案

> 版本：v1.0 | 状态：待确认
> 
> 核心问题：如何可靠地从「需要登录的网站」和「公开网页」采集知识数据？

---

## 一、采集需求分析

### 1.1 数据来源类型

| 来源类型 | 典型网站 | 技术难点 |
|---------|---------|---------|
| **需登录的内容平台** | 小红书、知乎、飞书、Notion | Cookie 维护、反爬机制、动态渲染 |
| **公开技术网站** | 技术博客、GitHub、官方文档 | 相对简单，httpx 即可 |
| **本地文件** | 导出的 Markdown / PDF / Word | 无网络问题，直接读取 |
| **RSS / API 接口** | 少量，按需支持 | 最简单，结构化数据 |

### 1.2 核心挑战

```
❌ 传统爬虫的问题：
  - 登录态维护难（Cookie 频繁过期）
  - 平台反爬检测（行为指纹识别）
  - 动态渲染（React/Vue SPA，httpx 拿不到内容）
  - 内容质量不稳定（爬到广告、无关内容）
  - 用户体验差（需要写复杂爬虫逻辑）

✅ 理想方案的要求：
  - 用户已登录的状态下自然采集
  - 支持选择性保存（用户控制质量）
  - 对目标网站友好（不触发反爬）
  - 支持批量和单篇采集
  - 采集结果可预览/编辑
```

## 二、三种方案对比

### 方案 A：传统 Python 爬虫（Playwright + Cookies）

```
用户手动导出 cookies.json
        ↓
Playwright 加载 Cookies 访问目标页
        ↓
提取正文内容
        ↓
保存到 raw/ 目录
```

**优点：**
- 纯后端，无需额外安装
- 可以定时批量采集
- 代码相对简单

**缺点：**
- Cookie 有效期短，频繁失效
- 平台检测 Playwright 特征（headless browser）
- 无法处理需要滑块验证码的场景
- 采集内容质量无法实时预览和过滤
- 对于小红书这类强反爬平台，稳定性差

**适合场景：** 公开技术文档、GitHub README 等简单场景

---

### 方案 B：浏览器插件（Chrome Extension）

```
用户在 Chrome 正常浏览目标网站
        ↓
点击插件图标 → 一键保存当前页
        ↓
插件提取页面内容 → 调用 API
        ↓
内容保存到知识库
```

**优点：**
- 无反爬问题（用户真实浏览器）
- 复用用户已登录状态
- 开发成本中等（Chrome Extension API）
- 不需要额外安装独立应用

**缺点：**
- 需要用户手动点击（不能批量自动）
- 无法做复杂的内容处理界面
- 需要额外开发和分发 .crx 插件文件
- 和后端通信需要处理 CORS

**适合场景：** 轻量级、偶尔使用的内容收藏

---

### 方案 C：Electron 独立采集应用（推荐 🌟）

```
┌─────────────────────────────────────────────────┐
│              Electron 采集应用                     │
│                                                 │
│  ┌──────────────┐    ┌─────────────────────┐   │
│  │  内嵌浏览器    │    │     控制面板          │   │
│  │ (Chromium)   │    │  ・采集队列           │   │
│  │              │    │  ・内容预览/编辑       │   │
│  │  用户正常登录  │    │  ・批量管理           │   │
│  │  浏览目标网站  │    │  ・目标KB选择         │   │
│  └──────┬───────┘    └──────────┬──────────┘   │
│         │   内容抓取              │  发送到后端   │
│         └────────────────────────┘              │
└─────────────────────────────────────────────────┘
                        ↓ HTTP API
              FastAPI 后端 → 处理流水线
```

**优点：**
- **零反爬问题**：真实 Chromium，用户手动登录，完全像人类操作
- **内嵌浏览器**：用户可以在应用内直接浏览目标网站
- **双栏界面**：左边浏览，右边管理采集队列和预览内容
- **多种采集模式**：手动保存单页 / 批量抓取指定 URL 列表 / 按规则自动捕获
- **内容预览**：采集前可预览、编辑，保证数据质量
- **本地文件支持**：直接拖拽导入 Markdown/PDF 文件
- **Cookie 持久化**：Electron 应用内维护登录状态，无需手动导出
- **可扩展**：可为不同网站写特定解析规则（Provider 机制）

**缺点：**
- 需要额外安装 Electron 应用（增加用户操作步骤）
- 开发工作量比纯 Python 方案更大
- 打包体积较大（~150MB）

**适合场景：** 这个项目的主要使用场景 ✅

## 三、推荐方案：Electron + FastAPI 分离架构

### 3.1 整体架构

```
┌─────────────────────┐         ┌─────────────────────┐
│   Electron 采集端    │  HTTP   │    FastAPI 后端       │
│                     │ ──────► │                     │
│  ・内嵌 Chromium     │         │  ・接收原始数据        │
│  ・采集控制面板       │         │  ・LLM 处理流水线      │
│  ・本地文件导入       │         │  ・Wiki 知识图谱       │
│  ・Cookie 管理       │         │  ・Milvus 向量库       │
└─────────────────────┘         └──────────┬──────────┘
                                           │
                                 ┌─────────▼──────────┐
                                 │    Vue 3 前端        │
                                 │  ・Wiki 浏览器       │
                                 │  ・搜索问答           │
                                 │  ・知识图谱           │
                                 │  ・流水线管理          │
                                 └─────────────────────┘
```

**部署关系：**
- Electron 应用：独立安装（macOS/Windows App）
- FastAPI + Milvus：本地 Docker 或云服务器
- Vue 前端：FastAPI 静态文件托管 或 独立 Nginx

### 3.3 核心技术设计

#### 采集模式

| 模式 | 触发方式 | 适用场景 |
|------|---------|---------|
| **手动单页保存** | 点击「保存当前页」按钮 | 边浏览边收藏 |
| **批量 URL 导入** | 粘贴多个 URL，批量抓取 | 已知需要采集哪些页面 |
| **本地文件导入** | 拖拽文件 | 已整理好的本地知识 |
| **自动捕获（可选）** | 设置规则，访问时自动保存 | 特定域名全量采集 |

#### 内容提取策略

```javascript
// 针对不同网站的 Provider 机制
const providers = {
  'xiaohongshu.com': XiaohongshuProvider,   // 小红书专用解析
  'zhihu.com':       ZhihuProvider,          // 知乎专用解析
  'default':         DefaultProvider,        // 通用提取（readability.js）
}

// 每个 Provider 实现:
interface ContentProvider {
  extract(webContents): RawDocument  // 提取正文
  getTitle(webContents): string       // 提取标题
  getMetadata(webContents): object    // 提取元数据（点赞数、作者等）
}
```

#### 与 FastAPI 后端通信

```
POST /api/collect/raw          # 提交单篇原始内容
POST /api/collect/batch        # 批量提交
GET  /api/kb/list              # 获取可用知识库列表
GET  /api/process/status/{id}  # 查询处理进度
```

#### Cookie 管理

```
Electron 应用使用独立 Session：
  - 各网站的 Cookie 自动持久化到 Electron userData 目录
  - 无需手动导出/导入 cookies.json
  - 用户在应用内登录一次，后续长期有效
  - 支持多账号切换（按 KB 区分 Session）
```

### 3.4 Electron 技术栈

| 模块 | 技术选型 | 说明 |
|------|---------|------|
| 框架 | Electron 32+ | 跨平台桌面应用 |
| 前端 | Vue 3 + Element Plus | 与主项目保持一致 |
| 构建 | Vite + electron-vite | 现代构建工具 |
| 打包 | electron-builder | 生成 .dmg / .exe |
| 内容提取 | readability.js + 自定义 Provider | 通用 + 专用解析 |
| HTTP 客户端 | axios | 与 FastAPI 通信 |
| 本地存储 | electron-store | 配置持久化 |

## 四、开发优先级建议

### Phase 1（MVP，先跑通整个流程）
- ✅ 本地文件导入（.md / .txt）
- ✅ 公开网页单页采集（无需登录）
- ✅ 基本的内容预览确认

### Phase 2（支持主要目标网站）
- 🔲 内嵌浏览器 + 手动登录
- 🔲 小红书专用 Provider
- 🔲 知乎专用 Provider
- 🔲 Cookie 自动管理

### Phase 3（效率提升）
- 🔲 批量 URL 采集
- 🔲 自动捕获规则
- 🔲 RSS 订阅支持
- 🔲 PDF 解析
