## 二、关键决策详解

### 2.1 LLM：Qwen via DashScope

**决策**：使用阿里云 DashScope API，通过 OpenAI 兼容接口调用

**接入方式**：
```python
from openai import AsyncOpenAI

client = AsyncOpenAI(
    api_key="YOUR_DASHSCOPE_API_KEY",
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1"
)

# LLM 调用
response = await client.chat.completions.create(
    model="qwen-plus",         # 主力模型
    messages=[...]
)

# Embedding 调用（同一个 client）
response = await client.embeddings.create(
    model="text-embedding-v3",
    input=["文本1", "文本2"]
)
```

**Qwen 模型选型参考**：

| 模型 | 用途 | 特点 |
|------|------|------|
| `qwen-max` | 复杂推理、实体概念抽取 | 最强效果，价格高 |
| `qwen-plus` | 主力：摘要生成、页面构建 | 效果/价格均衡 ✅ |
| `qwen-turbo` | 快速任务：清洗、判断 | 速度快，价格低 |
| `qwen-long` | 超长文档处理 | 最大 1M context |
| `text-embedding-v3` | 向量化 | 1024维，中英双语 ✅ |

**为什么不用 Claude（Anthropic）**：
- 目标是中文知识库场景，Qwen 对中文理解更好
- DashScope 在国内网络环境下更稳定
- 一个 API Key 同时支持 LLM + Embedding

### 2.2 向量数据库：Milvus Docker Standalone

**决策**：使用 Milvus Docker Standalone 部署

**原因**：
- 生产级性能，支持大规模向量检索
- pymilvus 2.4 的 `MilvusClient` 简化了 API
- 比 Weaviate 在国内生态和文档更成熟
- Docker 一键部署，运维简单

**连接配置**：
```python
from pymilvus import MilvusClient

client = MilvusClient(uri="http://localhost:19530")
```

**Collection 设计**（每个 KB 一个 Collection）：
```python
# Collection 名称：kb_{kb_id}
# 向量维度：1024（text-embedding-v3）
# 距离度量：COSINE
schema = {
    "fields": [
        {"name": "id",         "type": INT64,          "is_primary": True, "auto_id": True},
        {"name": "vector",     "type": FLOAT_VECTOR,   "dim": 1024},
        {"name": "doc_id",     "type": VARCHAR,        "max_length": 256},
        {"name": "page_type",  "type": VARCHAR,        "max_length": 32},   # source/entity/concept
        {"name": "title",      "type": VARCHAR,        "max_length": 512},
        {"name": "content",    "type": VARCHAR,        "max_length": 4096},
        {"name": "chunk_idx",  "type": INT64},
    ]
}
```

### 2.3 前端：Vue 3 + Vite + Element Plus

**决策**：Vue 3 + Vite（构建） + Element Plus（组件库）

**原因**：
- Element Plus 是 Element UI 的 Vue 3 版本，组件丰富，适合管理后台类界面
- Vite 构建速度快，开发体验好
- 与 electron-vite 生态完全兼容（后续 Electron 采集端可复用组件）

**UI 风格参考**：Obsidian（Wiki 浏览器部分）

**图谱可视化**：Apache ECharts（`echarts-for-vue`）
- 支持力导向图（Graph）
- 与 Element Plus 同属阿里系，风格统一
- 国内文档和社区资源丰富

**Markdown 渲染**：
- `markdown-it` 核心渲染
- 自定义插件处理 `[[WikiLink]]` → 可点击的内部链接
- `highlight.js` 代码高亮

---

### 2.4 数据采集：Electron 独立应用

**决策**：独立 Electron 应用，与 FastAPI 后端通过 HTTP 通信

**核心理由**：
- 目标网站（小红书等）有强反爬机制，传统 Playwright 不稳定
- Electron 内嵌真实 Chromium，用户手动登录后与正常浏览无异
- 采集内容可在应用内预览/编辑，保证数据质量

**技术栈**：
- 框架：Electron 32 + electron-vite
- 前端：Vue 3 + Element Plus（复用主项目组件）
- 内容提取：`@mozilla/readability`（Readability.js）+ 自定义 Provider
- 与后端通信：axios
- Cookie 管理：Electron 内置 Session（自动持久化，无需导出）

**是否独立仓库**：⚠️ **待决策**（见问题清单）

---

### 2.5 文件存储策略

**决策**：Wiki 知识图谱以 Markdown 文件存储

**原因**：
- 透明可读：用 Obsidian 也能直接打开查看
- 版本控制友好：可以用 Git 追踪知识变化
- 格式标准：`[[WikiLink]]` 是 Obsidian 通用格式

**目录结构**：
```
knowledge_bases/
└── {kb_id}/
    ├── config.json           # KB 配置
    ├── raw/                  # 原始采集内容（Markdown）
    ├── wiki/
    │   ├── source/           # 每篇文章的结构化摘要
    │   ├── entity/           # 实体页面（Claude、OpenAI等）
    │   ├── concept/          # 概念页面（agent、rag等）
    │   └── index.md          # 全局索引 + 别名
    └── db/
        ├── fts.db            # SQLite FTS5 全文索引
        └── graph.json        # 实体关系图
```

**注意**：Milvus 向量数据存在 Docker 容器的 volume 中，独立于文件目录。
